with associated payment records, user details, and coverage items. It worked fine in development. In production, under load, it was taking 3–8 seconds to respond.</p>\n<p>The culprit: a single <code>findOne</code> call with four nested <code>include</code> associations.</p>\n<p>Here is exactly what was happening, and how we fixed it.</p>\n<h2>The Original Code</h2>\n<pre><code class=\"language-typescript\">const policy = await Policy.findOne({\n  where: { id: policyId },\n  include: [\n    {\n      model: Payment,\n      where: { status: &#39;completed&#39; },\n    },\n    {\n      model: User,\n      attributes: [&#39;id&#39;, &#39;email&#39;, &#39;phone&#39;],\n    },\n    {\n      model: Coverage,\n      include: [\n        {\n          model: CoverageItem,\n        },\n      ],\n    },\n  ],\n});\n</code></pre>\n<p>Clean, readable, and completely standard Sequelize usage. The problem is invisible until you look at what SQL it actually generates.</p>\n<h2>What SQL Sequelize Actually Generates</h2>\n<p>Run your query with <code>logging: console.log</code> or check your database slow query log. Here is the simplified version of what Sequelize generates for <code>findOne</code> with multiple includes:</p>\n<pre><code class=\"language-sql\">SELECT\n  `Policy`.`id`,\n  `Policy`.`status`,\n  `Payment`.`id` AS `Payment.id`,\n  `Payment`.`status` AS `Payment.status`,\n  `User`.`id` AS `User.id`,\n  `User`.`email` AS `User.email`,\n  `Coverage`.`id` AS `Coverage.id`\nFROM (\n  SELECT `Policy`.`id`\n  FROM `policies` AS `Policy`\n  WHERE `Policy`.`id` = 42\n  LIMIT 1\n) AS `Policy`\n\nLEFT OUTER JOIN `payments` AS `Payment`\n  ON `Policy`.`id` = `Payment`.`policy_id`\n  AND `Payment`.`status` = &#39;completed&#39;\n\nLEFT OUTER JOIN `users` AS `User`\n  ON `Policy`.`id` = `User`.`policy_id`\n\nLEFT OUTER JOIN `coverages` AS `Coverage`\n  ON `Policy`.`id` = `Coverage`.`policy_id`\n</code></pre>\n<p>See that inner <code>SELECT</code> with <code>LIMIT 1</code> at the top? That is the problem.</p>\n<p>Sequelize wraps the primary table selection in a <strong>derived subquery</strong> before performing the joins. MySQL has to:</p>\n<ol>\n<li>Execute the inner subquery to get the matching <code>id</code></li>\n<li>Create a temporary derived table from that result</li>\n<li>Join every included model against that derived table</li>\n</ol>\n<p>Derived tables — unlike real tables — <strong>cannot use indexes on the join columns</strong>. MySQL has no statistics on them. With large associated tables (hundreds of thousands of payment rows, for example), the joins against an unindexed derived table become full scans.</p>\n<p>That is why your query is slow. MySQL can now use the foreign key indexes on <code>policy_id</code> for every join, and the <code>WHERE</code> clause filters on the primary key before any join work is done.</p>\n<p>In our case, this dropped the query from 3–8 seconds to under 50 milliseconds.</p>\n<h2>Why subQuery: false Works</h2>\n<p>When Sequelize uses <code>findOne</code> or <code>findAll</code> with a <code>LIMIT</code>, it wraps the primary table selection in a subquery to ensure the limit is applied before the joins — otherwise a <code>LIMIT 1</code> on a joined result set would not reliably give you exactly one <em>parent</em> record if the parent has multiple associated children.</p>\n<p>That is a legitimate concern. <code>subQuery: false</code> disables this protection. So there is a trade-off you need to understand:</p>\n<p><strong>When <code>subQuery: false</code> is safe:</strong></p>\n<ul>\n<li>You are filtering by a unique column (<code>id</code>, unique slug, etc.) that guarantees at most one parent row</li>\n<li>Your associations are many-to-one (each policy has one user, one payment record per query)</li>\n<li>You are applying <code>required: true</code> on includes that have strict one-to-one or one-to-many relationships you are filtering down</li>\n</ul>\n<p><strong>When <code>subQuery: false</code> is NOT safe:</strong></p>\n<ul>\n<li>You use <code>LIMIT 5</code> to get the &quot;top 5 policies&quot; and each policy has multiple payments — without the subquery, the LIMIT may cut across the joined rows and return fewer than 5 policies</li>\n<li>Your primary table <code>WHERE</code> does not uniquely identify a small set of rows</li>\n</ul>\n<p>For a typical <code>findOne</code>-equivalent (looking up a single record by ID), <code>subQuery: false</code> with <code>limit: 1</code> is always safe.</p>\n<h2>required: true — LEFT JOIN vs INNER JOIN</h2>\n<p>The second change is adding <code>required: true</code> on includes where you know the association always exists.</p>\n<p><code>required: false</code> (the default) generates a <code>LEFT OUTER JOIN</code>. MySQL must check every row in the left table and fill in NULL for missing matches. With large tables, this is expensive even when the data is almost always present.</p>\n<p><code>required: true</code> generates an <code>INNER JOIN</code>. MySQL only returns rows where the join condition matches. This is faster when the association is always expected to exist, because:</p>\n<ul>\n<li>The query engine can use the join condition to filter rows early</li>\n<li>The result set is smaller from the start</li>\n<li>The execution plan is simpler</li>\n</ul>\n<p>Use <code>required: false</code> only for truly optional associations — where a parent record can exist without children.

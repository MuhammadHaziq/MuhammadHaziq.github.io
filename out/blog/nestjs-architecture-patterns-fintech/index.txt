Its opinionated structure — modules, dependency injection, decorators — maps well onto the domain-driven design patterns that fintech applications need. But the framework gives you structure, not architecture. The decisions that matter are still yours to make.</p>\n<p>After building payment systems at InsuranceMarket.ae and FinanceThat, here are the patterns that have proven reliable in production fintech environments.</p>\n<h2>Module Organization: Domain-First, Not Layer-First</h2>\n<p>The instinct in many NestJS tutorials is to organize by technical layer: a <code>controllers/</code> folder, a <code>services/</code> folder, a <code>repositories/</code> folder. This works fine for small applications. It breaks down when the domain grows.</p>\n<p>Organize by domain instead:</p>\n<pre><code>src/\n  payments/\n    payments.module.ts\n    payments.controller.ts\n    payments.service.ts\n    payments.repository.ts\n    dto/\n    entities/\n    events/\n  policies/\n    policies.module.ts\n    ...\n  users/\n    users.module.ts\n    ...\n  shared/\n    database/\n    events/\n    guards/\n    interceptors/\n</code></pre>\n<p>Each domain module owns its own controller, service, and repository. Cross-domain communication happens through the <code>shared/events/</code> layer, not through direct service injection. This keeps modules independently testable and deployable.</p>\n<h2>Payment State Machines with Explicit Transitions</h2>\n<p>The biggest architectural mistake in payment systems is implicit state. When an order can be <code>pending</code>, <code>processing</code>, <code>paid</code>, <code>failed</code>, or <code>refunded</code>, every possible transition needs to be explicit — and transitions that don&#39;t make sense should be impossible.</p>\n<pre><code class=\"language-typescript\">// payments/payment-state-machine.ts\ntype OrderStatus = &#39;pending&#39; | &#39;processing&#39; | &#39;paid&#39; | &#39;failed&#39; | &#39;refunded&#39;;\n\nconst VALID_TRANSITIONS: Record&lt;OrderStatus, OrderStatus[]&gt; = {\n  pending:    [&#39;processing&#39;, &#39;failed&#39;],\n  processing: [&#39;paid&#39;, &#39;failed&#39;],\n  paid:       [&#39;refunded&#39;],\n  failed:     [],\n  refunded:   [],\n};\n\nexport function validateTransition(\n  from: OrderStatus,\n  to: OrderStatus\n): void {\n  if (!VALID_TRANSITIONS[from].includes(to)) {\n    throw new Error(`Invalid transition: ${from} → ${to}`);\n  }\n}\n</code></pre>\n<p>Calling <code>validateTransition</code> before every status update means that invalid state transitions fail loudly with a clear error, not silently with data corruption.</p>\n<p>In the service:</p>\n<pre><code class=\"language-typescript\">async processPayment(orderId: string): Promise&lt;void&gt; {\n  const order = await this.ordersRepo.findOneOrFail({ where: { id: orderId } });\n\n  validateTransition(order.status, &#39;processing&#39;);\n\n  await this.ordersRepo.update(\n    { id: orderId, status: &#39;pending&#39; },    // atomic transition\n    { status: &#39;processing&#39; }\n  );\n\n  // ... call gateway\n}\n</code></pre>\n<h2>Typed Error Handling with Domain Exceptions</h2>\n<p>NestJS&#39;s built-in HTTP exceptions are fine for simple APIs. For fintech, you need domain-level exceptions that carry enough context for logging and retry decisions.</p>\n<pre><code class=\"language-typescript\">// shared/exceptions/payment.exceptions.ts\nexport class PaymentGatewayException extends Error {\n  constructor(\n    public readonly gatewayName: string,\n    public readonly gatewayCode: string,\n    public readonly isRetryable: boolean,\n    message: string,\n  ) {\n    super(message);\n    this.name = &#39;PaymentGatewayException&#39;;\n  }\n}\n\nexport class InsufficientFundsException extends PaymentGatewayException {\n  constructor(gatewayName: string) {\n    super(gatewayName, &#39;INSUFFICIENT_FUNDS&#39;, false, &#39;Payment declined: insufficient funds&#39;);\n  }\n}\n\nexport class GatewayTimeoutException extends PaymentGatewayException {\n  constructor(gatewayName: string) {\n    super(gatewayName, &#39;TIMEOUT&#39;, true, &#39;Payment gateway timeout — safe to retry&#39;);\n  }\n}\n</code></pre>\n<p>The <code>isRetryable</code> flag is critical: it tells the calling code whether this failure should be retried or surfaced to the user immediately. A timeout is retryable. A declined card is not.</p>\n<p>An exception filter then handles these uniformly:</p>\n<pre><code class=\"language-typescript\">@Catch(PaymentGatewayException)\nexport class PaymentExceptionFilter implements ExceptionFilter {\n  catch(exception: PaymentGatewayException, host: ArgumentsHost) {\n    const ctx = host.switchToHttp();\n    const response = ctx.getResponse&lt;Response&gt;();\n\n    const statusCode = exception.isRetryable ? 503 : 402;\n\n    this.logger.error(&#39;Payment gateway error&#39;, {\n      gateway: exception.gatewayName,\n      code: exception.gatewayCode,\n      retryable: exception.isRetryable,\n    });\n\n    response.status(statusCode).json({\n      statusCode,\n      message: exception.message,\n      retryable: exception.isRetryable,\n    });\n  }\n}\n</code></pre>\n<h2>Interceptors for Consistent Response Shaping</h2>\n<p>Every API response should have a consistent envelope — status, data, and pagination metadata where applicable. A transform interceptor handles this once rather than in every controller method.</p>\n<pre><code class=\"language-typescript\">@Injectable()\nexport class TransformInterceptor&lt;T&gt;\n  implements NestInterceptor&lt;T, ApiResponse&lt;T&gt;&gt;\n{\n  intercept(\n    context: ExecutionContext,\n    next: CallHandler,\n  ): Observable&lt;ApiResponse&lt;T&gt;&gt; {\n    return next.handle().pipe(\n      map((data) =&gt; ({\n        success: true,\n        data,\n        timestamp: new Date().toISOString(),\n      })),\n    );\n  }\n}\n</code></pre>\n<p>Applied globally in <code>main.ts</code>:</p>\n<pre><code class=\"language-typescript\">app.useGlobalInterceptors(new TransformInterceptor());\n</code></pre>\n<h2>Observability: Structured Logging from Day One</h2>\n<p>The biggest mistake I see in early-stage fintech APIs is adding logging &quot;later.&quot; In payment systems, you need structured logs from day one — because when something goes wrong at 2am, the logs are what you have.</p>\n<p>Use NestJS&#39;s built-in logger with a structured format:</p>\n<pre><code class=\"language-typescript\">@Injectable()\nexport class PaymentsService {\n  private readonly logger = new Logger(PaymentsService.name);\n\n  async processPayment(orderId: string, gatewayName: string) {\n    this.logger.log(&#39;Payment initiated&#39;, {\n      orderId,\n      gateway: gatewayName,\n      timestamp: new Date().toISOString(),\n    });\n\n    try {\n      // ...\n      this.logger.log(&#39;Payment confirmed&#39;, { orderId, gateway: gatewayName });\n    } catch (error) {\n      this.logger.error(&#39;Payment failed&#39;, {\n        orderId,\n        gateway: gatewayName,\n        error: error.message,\n        stack: error.stack,\n      });\n      throw error;\n    }\n  }\n}\n</code></pre>\n<p>For production, pipe this to a structured logging service. At InsuranceMarket.ae we use Axiom. Papertrail and Datadog also work well. The key is that every log entry for a payment carries the <code>orderId</code> — which lets you reconstruct the complete lifecycle of any payment in seconds.</p>\n<h2>Health Checks for Each Gateway</h2>\n<p>Payment gateways go down. They go into maintenance mode. They return 200 OK with an error body. A health check endpoint that verifies each gateway is reachable saves you from discovering downtime via customer complaints.</p>\n<pre><code class=\"language-typescript\">@Controller(&#39;health&#39;)\nexport class HealthController {\n  constructor(\n    private readonly checkoutService: CheckoutService,\n    private readonly tapService: TapService,\n  ) {}\n\n  @Get()\n  async check() {\n    const [checkout, tap] = await Promise.allSettled([\n      this.checkoutService.ping(),\n      this.tapService.ping(),\n    ]);\n\n    return {\n      status: &#39;ok&#39;,\n      gateways: {\n        checkout: checkout.status === &#39;fulfilled&#39; ? &#39;up&#39; : &#39;down&#39;,\n        tap: tap.status === &#39;fulfilled&#39; ? &#39;up&#39; : &#39;down&#39;,\n      },\n    };\n  }\n}\n</code></pre>\n<p>Expose this endpoint to your monitoring system and page on it. A gateway being down is recoverable. Being down for 3 hours without knowing is not.</p>\n<h2>Testing Strategy</h2>\n<p>Unit tests for business logic (state machine, calculations, validation). Integration tests for the service layer against a real test database. No mocking the database in integration tests — that&#39;s how mock/prod divergence causes production incidents.</p>\n<p>For payment gateway calls, use their official sandbox environments rather than mocking.
